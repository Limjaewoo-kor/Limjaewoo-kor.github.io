Currently in progress

Issue

2 issues

1. 수십만개의 데이터를 가진 여러 테이블의 조인으로 인하여 쿼리 락 시간문제로 테이블스페이스 초과 및 컨테이너 다운 문제 발생.
2. 웹만 운영되던 사이트의 앱으로도 가능하도록 추가 요청 및 앱으로 진입시 특정 Function 수행.

Solution

2 solutions

1. 약 20만개의 물품데이터[15개이상의 테이블 조인]대한 상세 정보를 보여주는 화면이 있었습니다.
해당 화면의 쿼리는 화면에 따라 15초~60초 이상으로 큰 부하가 걸리는 상황이었으며,
해당 화면을 제약 분야에 종사하는 개인 및 기업들이 주기적으로 크롤링을 하는 상황이다보니 수시로 서버에 큰 부하가 걸렸습니다.
해당 사항으로 인해 서버에 TableSpace 용량 초과부터 간간히 컨테이너 다운 문제까지 지속적으로 발생하였고
"어떻게 부하를 줄일 수 있을까?" 고민끝에 생각해낸 방법이 바로 View의 캐시화였습니다.
Filter에서 Url로 분기를 태워서 해당 화면 자체를 파일로 캐시화하는 계획을 세웠고, ContentCachingResponseWrapper를 이용하여
랜더링된 html 파일 자체를 WAS내에 특정 경로에 저장하여서 이후에 해당 데이터에 변경이 있는 경우가 아니라면
저장된 파일을 사용하여 안정화시켰던 경험이 있습니다.

2. Progressive Web App을 사용하였던 상황인데 시각 장애인분들을 위한 요구사항으로 모바일 기기에서 특정 사이트에 접속하였을 경우,
메인 페이지에 " 웹사이트 바로가기" 아이콘을 생성할 수 있는 버튼을 만들고 해당 버튼으로 접속 시 메인 화면에 바코드를 인식하는 카메라 화면을
띄워주도록 하라는 요구사항이 있었습니다. 처음엔 "즐겨찾기 추가하는 기능을 구현하면 되겠구나"라고 생각하였으나 국내에서 엑티브x가 없어진 후에는
동의 없이 바로가기를 만드는 기술이 불가하게 되었고 대체하기위한 방법으로 PWA 기술의 a2hs 기능을 사용하면, 접속자의 동의는 필요하지만
웹 사이트의 바로가기 아이콘을 만들 수 있다는 것을 알게 되었습니다. PWA의 경우 익숙하지 않은 생소한 부분이다보니 처음엔 당황하였으나,
관련 레퍼런스를 참고하여 정리해보니 메니페스트파일과 로컬호스트 및 https의 url 그리고 serviceworker 등록까지 세가지만 있으면 된다는 걸 알게되었고,
개발 및 운영 환경에 적용하여 모바일 기기에 바로가기 앱을 구현하고 접속시 바코드를 인식하는 카메라 화면을 띄워주는 함수를 호출 하도록 하여 상위 기관의 요구사항을 충족시켰던 적이 있습니다.
